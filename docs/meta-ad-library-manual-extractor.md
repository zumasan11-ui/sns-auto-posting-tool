# Meta広告ライブラリ 手動表示カード抽出

Meta広告ライブラリをユーザーが手動で開き、検索・フィルター・スクロールした後に、画面上に表示済みの広告カードだけをCSV化する検証用ツールです。

## できること

- 表示されている広告カードから情報を抽出
- 掲載期間90日以上だけ残す
- 動画広告を除外する
- カルーセル広告、つまり複数枚クリエイティブの広告を除外する
- 掲載期間が長い順に並べる
- 広告ライブラリURLで重複除外しやすいCSVを作る
- CSVを画面表示し、可能ならクリップボードへコピー

## 取得項目

- 検索名
- 会社名
- 掲載開始日
- 掲載期間
- 広告ライブラリURL
- LP URL
- 広告本文
- 広告タイトル

## 使い方

1. ChromeでMeta広告ライブラリを開く
2. 検索名、国、active広告などを手動で指定する
3. 必要な件数までスクロールして広告カードを表示する
4. DevTools Consoleを開く
5. `scripts/meta_ad_library_visible_cards_extractor.js` の中身を貼り付けて実行する

検索名を指定したい場合は、貼り付け前にConsoleで次を実行します。

```js
window.META_AD_EXTRACTOR_OPTIONS = {
  searchName: "医療脱毛",
  minDurationDays: 90,
  existingUrls: []
};
```

実行後、右下にCSVテキストエリアが出ます。

## 重複除外

既存の広告ライブラリURLがある場合は、`existingUrls` に入れます。

```js
window.META_AD_EXTRACTOR_OPTIONS = {
  searchName: "医療脱毛",
  minDurationDays: 90,
  existingUrls: [
    "https://www.facebook.com/ads/library/?id=3425962427709054"
  ]
};
```

## Google Sheetsへ送る場合

ブラウザから直接Google Sheets APIに書くにはOAuthやCORSの面倒があります。最小構成では、Google Apps ScriptをWebアプリとして公開し、`sendToUrl` にそのURLを指定する方式が扱いやすいです。

```js
window.META_AD_EXTRACTOR_OPTIONS = {
  searchName: "医療脱毛",
  minDurationDays: 90,
  sendToUrl: "https://script.google.com/macros/s/xxxxx/exec"
};
```

送信ペイロードは次の形です。

```json
{
  "source": "meta_ad_library_visible_cards",
  "rows": []
}
```

## 検索履歴DB

同じスプレッドシート内の `検索履歴DB` は、AIが広告収集の検索状況を管理するための内部シートです。ユーザー入力用ではありません。

列:

- 検索名
- ジャンル
- サブジャンル
- 最終検索日
- 使用回数
- 90日以上候補数
- 新規追加数
- 重複スキップ数
- 失敗回数
- 状態
- 優先度
- メモ
- 最終ヒット日
- 次回検索予定日

`scripts/export_meta_search_terms.py` と `scripts/collect_meta_ads_today.py` はこの履歴を参照して、次回検索予定日が未来のワード、除外、一時停止のワードを検索対象から外します。対象になった検索名は、状態、優先度、次回検索予定日、最終検索日、最終ヒット日、使用回数、90日以上候補数、新規追加数、重複スキップ数、失敗回数を総合評価して並べます。

検索候補は単純なランダムではなく、履歴スコアで評価したうえでジャンルが偏らないように並び替えます。新規候補や未検索候補は早めに試し、成果が多い検索名は再検索間隔を短く、重複や失敗が多い検索名は間隔を長くします。

`scripts/append_meta_visible_rows_to_sheet.py` は広告収集結果を追記したあと、同じ検索名の履歴を更新します。新規追加があった時だけ `最終ヒット日` を更新し、成果に応じて `次回検索予定日` と `優先度` を自動更新します。

## 追加広告の分散ルール

`広告分析マスターDB` へ作成中で追加する広告は、掲載期間が長い順を基本にしつつ、同じジャンル、同じ会社、同じLP、同じサービス名に偏らないように選びます。

- 同じジャンルは原則2広告まで
- 同じ会社は原則1広告まで
- 同じLPは原則1広告まで
- 同じサービス名は原則1広告まで
- 広告が不足する場合だけ、ジャンル、会社、LP、サービス名の順に制限を段階的に緩和する
- 同じ広告ライブラリURLは必ず重複除外する

上限は環境変数または引数で変更できます。

- `DAILY_AD_GENRE_CAP`
- `DAILY_AD_COMPANY_CAP`
- `DAILY_AD_LP_CAP`
- `DAILY_AD_SERVICE_CAP`

## マスターDB直入れ運用

広告収集で見つけた広告は `広告分析マスターDB` に直接追加します。完成前の行は `分析状況/ステータス=作成中` にして行を黄色にし、作業中の広告が分かるようにします。

Notion日次ページは、Mac上でCodexに「リサーチして」と指示した時に `広告分析マスターDB` の作成中広告から1広告1ページで作ります。ユーザーが1広告1本の投稿本文と `訴求の型` を書いた広告だけ、投稿完了後に同じマスター行へ分析情報と投稿URLを追記し、黄色の背景色を消します。

## キーワードDBからの収集

通常は `scripts/run_today_ad_research_to_notion.py` を使います。この一発スクリプトは、カルーセル入れ替え、不足分収集、スクショURL補完、Notion日次ページ作成まで実行します。内部で使う `scripts/collect_meta_ads_today.py` は次の順に処理します。

1. `広告分析マスターDB` の作成中残数を数える
2. 目標件数に満たない場合だけ不足分を収集する
3. `キーワードDB` の横向き表から検索候補を読む。`キーワードDB` は1行目がジャンル、2行目以降が検索語の横型構造を維持する
4. `検索履歴DB` で除外、一時停止、優先度、次回検索予定日、最終検索日、最終ヒット日、使用回数、発見広告数、90日以上候補数、勝ち広告率、新規追加数、重複、失敗を見て検索順を最適化する
5. Meta広告ライブラリをPlaywrightで検索する
6. 90日以上かつ動画以外の広告を抽出する
7. `広告分析マスターDB` 内でURL重複を避ける
8. `広告分析マスターDB` に不足分だけ作成中で追加する
9. `検索履歴DB` を更新する
10. 会社名やサービス名から具体的な検索候補を `キーワードDB` へ追加する

1回のリサーチでは原則1広告をNotionへ投入します。`キーワードDB` は単なる単語リストではなく、Meta広告ライブラリ専用の検索エンジンとして育てます。検索価値の低い単語を増やすのではなく、勝ち広告の取りこぼしを減らすために、サービス名、ブランド名、商品名、有名企業名、具体的な悩み、施術名、機能名などを重複確認したうえで追加します。

広告取得後は、サービス名、会社名、広告本文、広告タイトル、画像コピー、LP URLから候補語を抽出します。`無料相談`、`資料請求`、`医療脱毛`、`勤怠管理` のような検索価値のある複合語を優先し、`安心`、`人気`、`おすすめ` など一般語やキャッチコピー断片は追加しません。

現在のDB構造、運用、検索品質を見て改善案がある場合は提案します。ただし、重複する項目や運用コストだけが増える提案は行わず、検索精度、分析精度、保守性が上がる場合だけ提案します。

通常の実行例:

```bash
python scripts/run_today_ad_research_to_notion.py \
  --chrome-executable "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
```

収集だけを個別実行する場合:

```bash
python scripts/collect_meta_ads_today.py \
  --search-limit 5 \
  --daily-target 1 \
  --chrome-executable "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
```

Codexへの合言葉は「リサーチして」。この指示が来たら、マスターDBの作成中不足分だけ広告リサーチを実行します。

## 掲載停止チェック（月1手動）

掲載停止チェックは毎日の広告リサーチ処理では実行しません。月1回、ユーザーがMacを開いて「停止リサーチして」と指示した時だけ `広告分析マスターDB` を確認します。

対象列:

- 状況
- 最終掲載期間

対象は、`分析日` から30日以上経っていて、`状況` が `掲載停止` ではない広告です。広告ライブラリURLを開いて、表示されていれば `掲載中`、止まっていれば `掲載停止` にします。

掲載停止の場合は、`掲載開始日` から確認日までを `最終掲載期間` に保存します。表示形式は広告収集時の掲載期間と同じく、`4ヶ月` / `1年2ヶ月` のように1ヶ月単位で切り捨てます。

## 注意

- Meta広告ライブラリのDOM変更に影響されます。
- 完全自動スクレイピングではなく、表示済みカードの手動補助抽出用です。
- カルーセル判定はMeta画面内の `carousel` / `カルーセル` / カード送り表示 / 複数クリエイティブのナビ要素から推定します。Meta側のDOM変更で漏れる場合があります。
- LP URLはカード内の外部リンクから推定します。Meta側の表示状態によって空欄になることがあります。
- 広告本文・タイトルは表示テキストからの推定です。
- 掲載期間は内部処理では日数を保持し、シート表示では `3ヶ月` / `1年2ヶ月` のように1ヶ月単位で切り捨て表示します。
- 会社名はMetaページ名を優先し、サービス名は広告文・広告タイトル・LPタイトルから推定します。URLやドメインだけを名前として保存しないようにしています。
