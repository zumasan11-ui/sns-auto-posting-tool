# プロジェクト運用ルール

このプロジェクトでは、実装だけでなく「作り方」も必ず残します。

## 基本方針

- チャット履歴が消えても、このプロジェクトだけで保守できる状態にする
- 新機能を追加したらREADMEまたはdocsを必ず更新する
- SNS投稿や認証まわりは、必要な環境変数と実行手順を明記する
- テンプレートの見た目を変えたら、どのファイルのどの役割かを記録する
- 生成物は原則 `deliverables/` に保存する
- 投稿ログやAPIレスポンス検証情報は `sns_posts/` に保存する
- 実装・検証・設定作業は基本的に自律的に進める。都度確認は不要で、確認するのは秘密情報の入力、OAuth認証、外部公開、課金・削除・不可逆操作など本人判断が本当に必要な時だけにする
- Notion本文の番号付きブロックは、奇数番号を広告分析、偶数番号をビジネスモデルとして扱う。画像が付くのは奇数番号の広告分析だけで、偶数番号のビジネスモデルには画像を付けない。偶数番号は画像が無くても、カルーセルと動画の両方でビジネスモデル用のテキストページを必ず1枚生成する
- Instagram/LinkedInのカルーセル画像は、1枚目を表紙、最後をプロフィール誘導にする。本文ページは `広告分析①`、`ビジネスモデル`、`広告分析②`、`ビジネスモデル` ... の順にし、最大 `広告分析④` までにする。広告1件なら4枚、広告2件なら6枚、広告3件なら8枚、広告4件なら10枚にし、空ページは追加しない。`広告分析メモ` や大きな `ビジネスモデル` 見出しは出さず、本文をそのまま見本寄せの細めゴシック体（Hiragino Sans W3優先）で表示する。本文が収まる場合は通常サイズと位置を維持し、溢れる時だけ文字サイズを下げて上下余白が均等になるよう中央寄せする
- SNS自動投稿は、PC電源OFF・スリープ中でもGitHub Actionsだけで何ヶ月も放置運用できる構成を最優先にする。一度OAuth/ログイン設定した後は、トークン期限切れや更新が必要なものを可能な限り自動更新し、更新後のトークンはGitHub Secretsへ書き戻す。手動更新が必要になる設計は最後の手段にする
- X、Threads、Facebookページのテキスト投稿は、広告分析/ビジネスモデルのセクションを朝 `07:30`、昼 `12:00`、夕 `16:00`、夜 `19:30` の4枠へ均等配分する。同じセクションは3媒体へ同じタイミングで投稿し、媒体ごとにはずらさない。枠内ではセクションごとに1分ずつずらす
- Instagram/LinkedInカルーセル投稿のキャプションとThreadsの日次投稿先頭は `広告分析` にする。Reels/Shortsなど動画系キャプションは `勝ち広告を分析してみました` のままにする
- カルーセル/動画のキャプションには、引用元が抽出できる場合は `引用元：...` を入れる。タイトル行と1つ目の引用元の間は空行を1行入れる
- Instagram Reelsは `share_to_feed=false` で投稿し、Instagramフィードには共有しない。Instagramフィード面はカルーセル投稿だけにする
- Reels/ShortsのBGMは固定1曲にしない。動画生成ごとにMixkitの無料BGM一覧からランダムに1曲取得して合成する。取得に失敗した場合だけ `assets/audio/mixkit_fallback/` の保存済み素材からランダムに選ぶ
- BGMの曲名・作者・URLを成果物メモや専用ログとして保存しない。生成物を増やしすぎないことを優先する
- Google Sheetsは1広告につき1行で保存する。広告分析本文と次のビジネスモデル本文を同じ行に入れ、掲載期間は保存しない。X投稿URLは広告分析側の1件だけをURL表示の `HYPERLINK` 式で保存する。ビジネスモデル側のX投稿URLは入れない
- 動画/カルーセルが2投稿目に分かれる場合、2投稿目の表紙はそのチャンク先頭の広告番号、つまり番号⑨などに書かれた掲載期間を使う
- Threadsは500文字を超える本文を1投稿に詰め込まず、500文字以内に分割して返信ツリーとして投稿する
- カルーセル/動画の本文は、通常サイズで収まる場合は従来のサイズと位置を維持する。収まらない場合だけ自動縮小し、最後を `...` で省略しない
- Facebook個人アカウントはAPI自動投稿対象外にする。代わりにReels生成時の動画とコピペ用キャプションを、Macローカル実行時は `~/Desktop/Facebook個人投稿用/`、GitHub Actions実行時は `deliverables/facebook_manual/` に毎回保存する。`latest` ファイルは作らない
- 自動投稿で増える `deliverables/auto_post/`、`public_state/public/runs/`、`public_state/public/manual_tests/`、Facebook個人手動投稿用フォルダのファイルは7日より古いものを自動削除する

## 新機能追加時に更新するもの

最低限、次を更新してください。

- `README.md`
- 関連する `docs/*.md`
- `.env.example` に必要な環境変数
- `requirements.txt` に必要な依存

## 書くべき内容

- 目的
- 入力
- 出力
- 実行コマンド
- 使用するAPI
- 必要な権限/スコープ
- 失敗しやすいポイント
- 将来拡張の方針

## 触ってはいけないもの

- `.env` の実値をREADME/docsに書かない
- 秘密情報はGitHub Secretsまたはローカル `.env` のみに保存する
- `.env`、`credentials/`、JSON秘密鍵、トークン類は絶対にGit管理しない
- README、docs、ログ、GitHub Actions出力に秘密情報を表示しない
- `.env.example` にはプレースホルダーだけを書く
- APIキー、アクセストークン、クライアントシークレット、JSONキーを扱う処理では値をマスクまたは非表示にする
- コミット前に `python3 scripts/secret_scan.py --mode staged` を実行する。初回は `bash scripts/install_git_hooks.sh` でpre-commit hookを入れる
- GitHub ActionsではSecretsから環境変数を読み込む構成にし、コードやworkflowへ実値を書かない
- APIキー、アクセストークン、クライアントシークレット、JSON秘密鍵の取得、`.env` への保存、GitHub Secretsへの反映はCodexが担当する
- 秘密情報を取得・保存・登録するときも、チャット、README、docs、ログ、Actions出力には実値を表示しない
- ユーザー本人の操作が必要なのは、ログイン、OAuth認証、利用規約同意、本人確認、権限付与など本人しか実行できない操作だけにする
- ユーザーが作った生成物を勝手に削除しない
- 既存テンプレートを壊す大きな変更を、互換性なしで入れない
