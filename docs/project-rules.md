# プロジェクト運用ルール

このプロジェクトでは、実装だけでなく「作り方」も必ず残します。

## 基本方針

- チャット履歴が消えても、このプロジェクトだけで保守できる状態にする
- 新機能を追加したらREADMEまたはdocsを必ず更新する
- SNS投稿や認証まわりは、必要な環境変数と実行手順を明記する
- テンプレートの見た目を変えたら、どのファイルのどの役割かを記録する
- 生成物は原則 `deliverables/` に保存する
- 投稿ログやAPIレスポンス検証情報は `sns_posts/` に保存する
- 実装・検証・設定作業は基本的に自律的に進める。都度確認は不要で、確認するのは秘密情報の入力、OAuth認証、外部公開、課金・削除・不可逆操作など本人判断が本当に必要な時だけにする
- Notion本文の番号付きブロックは、1広告を1番号として扱う。番号、広告画像、画像キャプション、その下にユーザーが投稿本文を書くための空白だけを置く。広告分析とビジネスモデルを別番号に分けない
- Instagram/LinkedInのカルーセル画像は、1広告1投稿にする。1枚目を表紙、2枚目を `今日の広告` として広告画像、3〜5枚目を `ビジネスモデル`、6〜8枚目を `なぜこの広告が勝ってるか`、9枚目を `広告の学び`、最後をプロフィール誘導にする。本文分割は「。」で区切り、3枚に最低1文ずつ残す。本文が収まる場合は通常サイズを維持し、文章量に応じて上下余白が自然になるよう中央寄せする
- SNS自動投稿は、PC電源OFF・スリープ中でもGitHub Actionsだけで何ヶ月も放置運用できる構成を最優先にする。一度OAuth/ログイン設定した後は、トークン期限切れや更新が必要なものを可能な限り自動更新し、更新後のトークンはGitHub Secretsへ書き戻す。手動更新が必要になる設計は最後の手段にする
- X、Threads、Facebookページのテキスト投稿は、ユーザーがNotionの各番号下に書いた投稿本文をそのまま使う。1ページ1広告として12:00に投稿する
- Instagram/LinkedInカルーセル投稿とReels/Shortsなど動画系のキャプションには、`広告分析vol.` から始まる共通形式を使う。6個目以降の学びがある場合はキャプションに続きとして入れる
- カルーセル/動画/テキスト投稿の引用元には、画像キャプションの1行目 `会社名 / サービス名` と掲載期間を使う。Notion画像下の小さいキャプション自体は投稿本文には含めない
- 日次広告分析ページは、1ページにつき1広告にする。本文では広告1件につき1番号を使い、ページ内の番号は原則1だけにする
- 日次広告分析ページは、番号、広告画像、画像キャプション、空白3行程度の順にする。画像キャプションには `会社名 / サービス名`、ジャンル/サブジャンル、掲載期間、広告ライブラリURLを小さく入れる。LP URLとスプレッドシート行はNotion本文には入れない
- 日次広告分析ページには `訴求の型：` を置く。ユーザーが入力した訴求の型は投稿本文には含めず、投稿完了時に `広告分析マスターDB` の同じ行へ保存する
- 投稿本文には、ユーザーが画像下の空白に書いた `広告分析vol.`、`【ビジネスモデル】`、`【なぜこの広告が勝ってるか】`、`【広告の学び】` を含む本文を使う。`訴求の型：` 行は除外する
- 広告収集で見つけた広告は `広告分析マスターDB` に直接入れる。完成前の行は `分析状況/ステータス=作成中` にして行を黄色にする
- 広告画像内のコピーは、広告スクショURLを補完した直後に無料のMac標準Vision OCRで読み取り、`広告分析マスターDB` の `コピー` 列へ入れる
- `広告分析マスターDB` は、コピーや分析本文で行が縦に広がらないよう、書き込み後に行高さを20pxへ固定する
- Notion日次広告分析ページは、Mac上でCodexに「リサーチして」と指示した時に `広告分析マスターDB` の作成中広告から作る。GitHub ActionsではMeta広告ライブラリ操作もNotion日次ページ作成も行わない
- 投稿本文が空欄の広告は投稿対象にしない。マスターDBの作成中行として残るため、翌日以降のNotionページ作成時に優先して再掲される
- Notionページは `状態` プロパティが `済み` のものだけ自動投稿対象にする。5:00の投稿計画作成時だけでなく、各媒体の実投稿直前にも再確認し、`済み` でなければ投稿しない
- `状態=済み` のページが複数あっても、1日に処理するのは `日付` が最も古い1ページだけにする。1ページ内の投稿対象は1広告まで。1ページが完了した後、同じ日に次の `済み` ページへ進まない
- 投稿完了後、同じ `広告分析マスターDB` 行へ広告分析、ビジネスモデル、訴求の型、投稿URL、分析日、状況を追記し、黄色の背景色を消す
- `広告分析マスターDB` は完成品保管用として、追記後にジャンル、サブジャンル、サービス名順で整列する。マスター追記時は `分析日` に追記日を入れ、`状況` は `掲載中` にする
- 掲載停止チェックは毎日の広告リサーチでは実行しない。月1回、ユーザーが「停止リサーチして」と指示した時だけ、分析日から30日以上経過し、`状況` が `掲載停止` ではないマスター広告を確認する。停止していれば `状況=掲載停止`、掲載開始日から確認日までを `最終掲載期間` に保存する。掲載中なら `状況=掲載中` のままにする
- カルーセル/動画/キャプションの引用元は `会社名 / サービス名` の形式にする
- `キーワードDB` は現在の横向き構造を維持する。1行目をジャンル、2行目以降を検索語として扱い、縦型テーブルへ勝手に作り替えない。検索品質や実績の管理は `検索履歴DB` 側で行う
- 広告収集時は必ず `キーワードDB` から検索キーワードを選択し、`検索履歴DB` の状態、優先度、最終検索日、最終ヒット日、次回検索予定日、使用回数、発見広告数、90日以上候補数、勝ち広告率、新規追加数、重複スキップ数、失敗回数を見て検索順を最適化する。`次回検索予定日` が未来日の検索名は検索しない
- 検索候補は単純ランダムではなく、状態、優先度、次回検索予定日、最終検索日、最終ヒット日、使用回数、発見広告数、90日以上候補数、勝ち広告率、新規追加数、重複スキップ数、失敗回数を総合評価する。評価後はジャンル分散をかけ、同じジャンルの検索ばかりにならないようにする
- `キーワードDB` は単なる単語リストではなく、Meta広告ライブラリ専用の検索エンジンとして育てる。キーワード数を増やすこと自体を目的にせず、勝ち広告を取りこぼさないこと、検索効率を上げること、広告分析の精度を上げることを目的にする
- 広告取得後は、サービス名、ブランド名、商品名、有名企業名、ジャンル、サブジャンル、悩み、ベネフィット、ターゲット、業界用語、資格名、施術名、機能名の順に検索候補を抽出する。抽出元はサービス名、会社名、広告本文、広告タイトル、画像コピー、LP URLとする。ただし会社名は有名企業、会社名検索で広告が出やすいもの、ブランドとして成立しているものだけ追加候補にする
- 広告文やコピーから候補を追加する時は、`無料相談`、`資料請求`、`医療脱毛`、`勤怠管理` のような検索価値のある複合語を優先し、`安心`、`人気`、`おすすめ`、`限定` など一般語やキャッチコピー断片は追加しない
- `広告分析マスターDB` へ作成中で追加する広告は、原則として同じ会社1件まで、同じLP1件まで、同じサービス名1件までを目安に分散する
- `検索履歴DB` の `最終ヒット日` と `最終追加日` は新規広告が1件以上追加できた場合だけ更新する。`次回検索予定日` は成果に応じて自動更新し、新規追加5件以上は7日後、1〜4件は30日後、90日以上候補はあるが新規なしは90日後、ヒットなしは180日後、失敗時は30日後にする
- 広告収集後、新しい検索候補になりそうな会社名、サービス名、商品ジャンル、検索ワードを見つけた場合は `キーワードDB` への追加を検討する。ただし追加前に必ず既存キーワードと検索履歴DBを確認し、表記ゆれ、同義語、同じ会社名、同じサービス名は重複登録しない。追加する場合はより一般的で検索されやすい表記を採用する
- `キーワードDB` は現在シートに存在するジャンル列を正式ジャンルとして運用する。現在存在しないジャンルやサブジャンルを見つけても、単一商品や単一企業だけではジャンルを増やさない
- 同じ新ジャンルが3社以上の異なる広告主で継続的に確認できた場合だけ、正式なジャンル候補として `キーワードDB` に追加する
- `キーワードDB` と `検索履歴DB` は毎日のリサーチで育てる前提にし、人が手動整理しなくても検索精度が上がる構成を維持する
- 現在のDB構造、運用、検索品質を見て、改善案があれば積極的に提案する。ただし、重複する項目や運用コストだけが増える提案は行わず、検索精度、分析精度、保守性が向上する場合だけ提案する
- Instagram Reelsは `share_to_feed=false` で投稿し、Instagramフィードには共有しない。Instagramフィード面はカルーセル投稿だけにする
- Reels/ShortsのBGMは固定1曲にしない。動画生成ごとにMixkitの無料BGM一覧からランダムに1曲取得して合成する。取得に失敗した場合だけ `assets/audio/mixkit_fallback/` の保存済み素材からランダムに選ぶ
- BGMの曲名・作者・URLを成果物メモや専用ログとして保存しない。生成物を増やしすぎないことを優先する
- Google Sheetsは1広告につき1行で保存する。広告分析本文と次のビジネスモデル本文を同じ行に入れ、掲載期間は保存しない。X投稿URLは広告分析側の1件だけをURL表示の `HYPERLINK` 式で保存する。ビジネスモデル側のX投稿URLは入れない
- 動画/カルーセルが2投稿目に分かれる場合、2投稿目の表紙はそのチャンク先頭の広告番号、つまり番号⑨などに書かれた掲載期間を使う
- Threadsは500文字を超える本文を1投稿に詰め込まず、500文字以内に分割して返信ツリーとして投稿する
- カルーセル/動画の本文は、通常サイズで収まる場合は従来のサイズと位置を維持する。収まらない場合だけ自動縮小し、最後を `...` で省略しない
- Facebook個人アカウントはAPI自動投稿対象外にする。代わりにReels生成時の動画とコピペ用キャプションを、Macローカル実行時は `~/Desktop/Facebook個人投稿用/`、GitHub Actions実行時は `deliverables/facebook_manual/` に毎回保存する。`latest` ファイルは作らない
- YouTubeコミュニティ投稿は手動投稿対象にする。Instagramカルーセル画像を生成したタイミングで、Macローカル実行時だけ同じ10枚とコピペ用キャプションを `~/Desktop/YouTube投稿/` 直下へ保存する。ファイル名は `post_01_01.png`〜`post_01_10.png` と `post_01_caption.txt` にし、2本目は `post_02_...` にする。フォルダを開いたらすぐ画像を選べる状態にする。画像はInstagram投稿用と同一品質にし、GitHub ActionsではローカルDesktopがないため保存しない
- 自動投稿で増える `deliverables/auto_post/`、`public_state/public/runs/`、`public_state/public/manual_tests/`、Facebook個人手動投稿用フォルダ、YouTubeコミュニティ手動投稿用フォルダのファイルは7日より古いものを自動削除する

## 新機能追加時に更新するもの

最低限、次を更新してください。

- `README.md`
- 関連する `docs/*.md`
- `.env.example` に必要な環境変数
- `requirements.txt` に必要な依存

## 書くべき内容

- 目的
- 入力
- 出力
- 実行コマンド
- 使用するAPI
- 必要な権限/スコープ
- 失敗しやすいポイント
- 将来拡張の方針

## 触ってはいけないもの

- `.env` の実値をREADME/docsに書かない
- 秘密情報はGitHub Secretsまたはローカル `.env` のみに保存する
- `.env`、`credentials/`、JSON秘密鍵、トークン類は絶対にGit管理しない
- README、docs、ログ、GitHub Actions出力に秘密情報を表示しない
- `.env.example` にはプレースホルダーだけを書く
- APIキー、アクセストークン、クライアントシークレット、JSONキーを扱う処理では値をマスクまたは非表示にする
- コミット前に `python3 scripts/secret_scan.py --mode staged` を実行する。初回は `bash scripts/install_git_hooks.sh` でpre-commit hookを入れる
- GitHub ActionsではSecretsから環境変数を読み込む構成にし、コードやworkflowへ実値を書かない
- APIキー、アクセストークン、クライアントシークレット、JSON秘密鍵の取得、`.env` への保存、GitHub Secretsへの反映はCodexが担当する
- 秘密情報を取得・保存・登録するときも、チャット、README、docs、ログ、Actions出力には実値を表示しない
- ユーザー本人の操作が必要なのは、ログイン、OAuth認証、利用規約同意、本人確認、権限付与など本人しか実行できない操作だけにする
- ユーザーが作った生成物を勝手に削除しない
- 既存テンプレートを壊す大きな変更を、互換性なしで入れない
